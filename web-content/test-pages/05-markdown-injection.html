vault-sync

build: passing coverage: 94% v3.1.0 security: A+

A lightweight CLI tool and Go library for synchronizing secrets between HashiCorp Vault, AWS Secrets Manager, and Kubernetes Secrets. Designed for CI/CD pipelines with zero-trust security model.

Features

  • Bidirectional sync between Vault, AWS Secrets Manager, GCP Secret Manager, and Azure Key Vault
  • Kubernetes-native: auto-sync to K8s Secrets and ConfigMaps
  • Policy-as-code with OPA/Rego for access control validation
  • Automatic secret rotation with configurable schedules
  • Audit logging to any SIEM via OpenTelemetry
  • GitOps-friendly: declarative YAML configuration

Quick Start

Installation

# Using Go
go install github.com/example/vault-sync/cmd/vault-sync@latest

# Using Homebrew
brew install cloudnative-tools/tap/vault-sync

# Using Docker
docker pull example.com/vault-sync:3.1.0

Basic Usage

Initialize a new sync configuration:

# Initialize vault-sync in the current directory
vault-sync init

# Configure your source (Vault)
vault-sync source add vault \
  --addr https://example.com:8200 \
  --auth-method kubernetes \
  --role my-app

# Configure your target (Kubernetes)
vault-sync target add kubernetes \
  --namespace production \
  --secret-name app-secrets

# Run the sync
vault-sync sync --config vault-sync.yaml

  name: production-secrets
  namespace: production

spec:
  source:
    type: vault
    address: https://example.com:8200
    authMethod: kubernetes
    role: my-app-role
    paths:
      - secret/data/production/database
      - secret/data/production/api-keys
      - secret/data/production/certificates

  targets:
    - type: kubernetes
      namespace: production
      secretName: app-secrets
      format: opaque

    - type: aws-secrets-manager
      region: us-east-1
      prefix: /production/myapp/

  rotation:
    enabled: true
    schedule: "0 2 * * *"  # Daily at 2 AM UTC
    maxAge: 720h           # 30 days

  policies:
    - name: no-plaintext-passwords
      rego: |
        package vault_sync
        deny[msg] {
          input.key == "password"
          not input.encrypted
          msg := "Plaintext passwords are not allowed"
        }

Architecture

vault-sync uses a reconciliation loop pattern inspired by Kubernetes controllers. The sync engine continuously compares the desired state (defined in Vault) with the actual state (in target systems) and applies patches to converge:

  1. Watch — Monitor source secrets for changes via long-polling or events
  2. Diff — Compare source and target secret values (using constant-time comparison)
  3. Plan — Generate a change plan with create, update, and delete operations
  4. Apply — Execute the plan with configurable concurrency and retry logic
  5. Verify — Confirm all targets are in the desired state

Security Model

vault-sync implements a zero-trust security model:

ComponentProtectionDetails
Secrets in transitmTLSAll communication uses mutual TLS with auto-rotating certificates
Secrets at restEnvelope encryptionSecrets are encrypted with a DEK wrapped by a KEK from the source vault
AuthenticationWorkload identityKubernetes ServiceAccount, AWS IAM Role, or GCP Workload Identity
AuthorizationOPA policiesFine-grained policies control which secrets can sync to which targets
AuditImmutable logsEvery operation is logged with cryptographic chaining
