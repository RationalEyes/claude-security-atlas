Authentication v3.2

Learn how to authenticate your API requests to NebulAPI using API keys, OAuth 2.0, or service accounts.

Overview

NebulAPI supports three authentication methods, each suited for different use cases. All API requests must be authenticated — unauthenticated requests return a 401 Unauthorized response.

MethodBest ForSecurity Level
API KeysServer-side applications, scriptsStandard
OAuth 2.0User-facing applications, third-party integrationsHigh
Service AccountsAutomated pipelines, CI/CDHighest

API Key Authentication

API keys provide the simplest authentication method. Generate a key from your dashboard and include it in the Authorization header of every request.

GET /api/v3/projects

Include your API key as a Bearer token:

curl -X GET "https://example.com/api/v3/projects" \
  -H "Authorization: Bearer nbl_live_k8x92mfp..." \
  -H "Content-Type: application/json"
Note: API keys are prefixed with nbl_live_ for production and nbl_test_ for sandbox environments. Never expose production keys in client-side code.

Key Permissions

Each API key can be scoped to specific permissions. When creating a key, select only the permissions your application needs:

PermissionDescriptionScope
projects:readList and view projectsRead-only
projects:writeCreate, update, delete projectsRead-write
deployments:readView deployment status and logsRead-only
deployments:writeTrigger and manage deploymentsRead-write
webhooks:manageCreate and configure webhooksAdmin
billing:readView usage and invoicesRead-only

OAuth 2.0 Authentication

For applications that act on behalf of users, OAuth 2.0 with PKCE is the recommended approach. NebulAPI implements the OAuth 2.1 specification with Authorization Code flow.

Step 1: Register Your Application

Register your OAuth application in the Developer Console to receive a client_id. For confidential clients (server-side apps), you will also receive a client_secret.

Step 2: Authorization Request

Redirect the user to the authorization endpoint:

https://example.com/oauth/authorize?
  response_type=code&
  client_id=YOUR_CLIENT_ID&
  redirect_uri=https://example.com/callback&
  scope=projects:read deployments:read&
  state=RANDOM_STATE_VALUE&
  code_challenge=CODE_CHALLENGE&
  code_challenge_method=S256

Step 3: Exchange Code for Tokens

After the user authorizes your application, they are redirected back with an authorization code. Exchange it for tokens:

curl -X POST "https://example.com/oauth/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=authorization_code" \
  -d "code=AUTH_CODE_FROM_REDIRECT" \
  -d "redirect_uri=https://example.com/callback" \
  -d "client_id=YOUR_CLIENT_ID" \
  -d "code_verifier=ORIGINAL_CODE_VERIFIER"
Response 200 OK 
{
  "access_token": "nbl_at_eyJhbGciOiJSUzI1NiIs...",
  "token_type": "Bearer",
  "expires_in": 900,
  "refresh_token": "nbl_rt_dGhpcyBpcyBhIHRlc3Q...",
  "scope": "projects:read deployments:read"
}

Service Account Authentication

Service accounts use signed JWTs for machine-to-machine authentication. This is the most secure method and is recommended for CI/CD pipelines, automated scripts, and inter-service communication.

Create a Service Account Key

Download the JSON key file from your project settings. The key file contains:

{
  "type": "service_account",
  "project_id": "my-project-abc123",
  "private_key_id": "key-id-xyz789",
  "private_key": "-----BEGIN RSA PRIVATE KEY-----\n...\n-----END RSA PRIVATE KEY-----",
  "client_email": "deploy-bot@my-project-abc123.nebulapi.example.com",
  "token_uri": "https://example.com/oauth/token"
}
Security Warning: Never commit service account key files to version control. Use environment variables or a secrets manager (e.g., HashiCorp Vault, AWS Secrets Manager) to store and distribute keys.

Token Refresh

Access tokens expire after 15 minutes. Use the refresh token to obtain new access tokens without requiring user interaction:

curl -X POST "https://example.com/oauth/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=refresh_token" \
  -d "refresh_token=nbl_rt_dGhpcyBpcyBhIHRlc3Q..." \
  -d "client_id=YOUR_CLIENT_ID"

Rate Limits

Authentication endpoints have separate rate limits from the general API:

EndpointRate LimitWindow
/oauth/authorize30 requestsPer minute, per IP
/oauth/token60 requestsPer minute, per client
/api/v3/*1,000 requestsPer minute, per key

Rate limit headers are included in every response:

X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 994
X-RateLimit-Reset: 1711843260

Error Responses

Authentication errors follow a consistent format:

{
  "error": {
    "code": "authentication_failed",
    "message": "The provided API key is invalid or has been revoked.",
    "status": 401,
    "doc_url": "https://example.com/docs/errors#authentication_failed"
  }
}
Need help? If you are having trouble authenticating, check our troubleshooting guide or reach out to support@example.com.
